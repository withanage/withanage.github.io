11 * @file classes/metadata/MetadataSchema.php
12 *
13 * Copyright (c) 2014-2021 Simon Fraser University
14 * Copyright (c) 2000-2021 John Willinsky

16 *
17 * @class MetadataSchema
18 *
19 * @ingroup metadata
20 *
21 * @see MetadataProperty
22 * @see MetadataRecord
23 *
24 * @brief Class that represents a meta-data schema (e.g. NLM element-citation,
25 *  OpenURL, dc(terms), MODS) or a subset of it.
26 *
27 *  We only implement such subsets of meta-data schemas that contain elements which
28 *  can be mapped to PKP application objects. Meta-data schemas are not meant to
29 *  represent any meta-data in the given schema just PKP application meta-data. The
30 *  constructor argument uniquely identifies the application objects this meta-data
31 *  schema can be mapped to. There should never be two MetadataSchemas with the same
32 *  namespace that map to the same application object type. This also means that we
33 *  implement composite elements if and only if the composite complies with our
34 *  internal class composition schema and not only because the schema allows a composite
35 *  in a certain position. See MetadataDescription and MetadataProperty for further
36 *  information about composite meta-data properties.
37 *
38 *  Example: We implement a composite to represent authors that correspond to the
39 *  \PKP\author\Author class. We do not implement composites for title meta-data
40 *  even if the chosen schema allows this (e.g. abbreviated title, alternative title)
41 *  as this data is implemented as fields of the Submission object. This doesn't mean
42 *  that such data cannot be mapped to composites in external bindings, e.g. in an
43 *  XML binding of the meta-data schema. We can always map a flat list of key/value
44 *  pairs to a hierarchical representation in an external binding.
45 *
46 *  This coupling allows us to flexibly configure meta-data entry for application
47 *  objects. We can identify appropriate meta-data fields for application objects
48 *  when displaying or entering object meta-data in application forms. Thereby users
49 *  can dynamically add meta-data fields for input/output if they require these for
50 *  their local meta-data partners (e.g. libraries, repositories, harvesters, indexers).
51 *
52 *  We assume that all properties defined within a schema can potentially be assigned
53 *  to the objects represented by the given association types. Users should, however,
54 *  be able to enable / disable properties on a per-assoc-type basis so that only a
55 *  sub-set of properties will actually be available in the user interface as well as
56 *  exported or imported for these objects.
57 *
58 *  New schemas can be dynamically added to the mix at any time if they provide fields
59 *  not provided by already existing schemas.
60 *
61 *  NB: We currently provide meta-data schemas as classes for better performance
62 *  and code readability. It might, however, be necessary to maintain meta-data
63 *  schemas in the database for higher flexibility and easier run-time configuration/
64 *  installation of new schemas.
65 */
